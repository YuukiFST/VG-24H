1. O que fazer?

O projeto consiste no desenvolvimento de uma aplicação web para a Prefeitura de Várzea Grande, inspirada no aplicativo "Cuiabá Smart", executada exclusivamente como um portal responsivo em conformidade com o Design System do Governo Federal (Gov.br).

O sistema centraliza o recebimento de solicitações de serviços públicos, permitindo que o cidadão reporte problemas urbanos como buracos, falhas na iluminação pública, problemas de saneamento e acompanhe o andamento dos chamados pelo número de protocolo.

O banco de dados em PostgreSQL armazena e integra os dados, garantindo integridade e rastreabilidade por meio de Triggers e Rules implementados diretamente no banco.

2. Justificativa

2.1 Importância para a área proposta

A digitalização dos serviços públicos é essencial para a modernização da gestão municipal. Um portal web facilita o acesso dos cidadãos, democratiza a informação e permite que a prefeitura mapeie problemas recorrentes por meio de dados estatísticos.

2.2 Importância para o curso

  • Modelagem de dados relacional com entidades bem definidas
  • Controle de acesso por perfil de usuário (Cidadão, Colaborador, Administrador)
  • Desenvolvimento full-stack com Django e PostgreSQL
  • Recursos avançados de banco de dados: Triggers, Rules e Views

3. Funcionalidades e Requisitos

3.1 Autenticação e controle de acesso

  • Login padronizado (Gov.br) para Cidadão, Colaborador e Administrador
  • Cidadão faz seu cadastro pela plataforma
  • Contas de Colaborador criadas exclusivamente pelo Administrador
  • Recuperação de senha com token temporário
  • Senha temporária para Colaborador com troca obrigatória no 1º acesso

Módulo do Cidadão

  • Abertura de chamados: categoria → serviço → descrição → foto → bairro
  • Número do endereço obrigatório (informar 0 se não houver)
  • Adição de fotos e observações (bloqueado em CO/CA)
  • Cancelamento do próprio chamado (status AB e EA)
  • Acompanhamento via protocolo numérico com histórico de status
  • Avaliação do atendimento (nota 1-5, uma única vez)
  • Visualização e exclusão de notificações

Módulo do Colaborador

  • Dashboard (Semáforo) com listagem filtrável por bairro, status e data
  • Alteração de status — exceto reabrir chamados CO ou CA
  • Encerramento (CO/CA) exige campo de resolução obrigatório
  • Adição de observações e fotos como comprovação

Módulo do Administrador

  • Todas as funcionalidades do Colaborador
  • Alteração de status de qualquer chamado, incluindo CO e CA
  • Invalidação de chamados (status CA) com motivo em resolução
  • Gerenciamento de usuários, categorias, serviços e bairros
  • Acesso exclusivo ao painel de estatísticas consolidadas

3.2 Requisitos técnicos e de banco de dados

  • Chaves primárias (SERIAL), estrangeiras, NOT NULL, UNIQUE, CHECK
  • Protocolo numérico reiniciando a cada ano (ex: 20260001)
  • Registros inativados com campo ativo — nenhuma exclusão real
  • Fotos armazenadas no Cloudinary com compressão automática
  • Carga inicial: 2 categorias, 5 serviços, 5 status, bairros de VG

3.3 Recursos avançados de banco de dados

Triggers

  • Trigger 1 — AFTER INSERT em chamado: insere automaticamente o primeiro histórico com status AB
  • Trigger 2A — BEFORE UPDATE em chamado: atualiza atualizado_em
  • Trigger 2B — AFTER INSERT em historico_chamado: atualiza dt_conclusao e gera notificação

Triggers de Integridade (convertidos de Rules)

  • trg_foto_chamado_encerrado — bloqueia foto se chamado CO/CA
  • trg_historico_obs_encerrado — bloqueia observação se chamado CO/CA
  • trg_avaliacao_imutavel — impede reavaliação

Rules

  • R5 — impede UPDATE e DELETE em historico_chamado
  • Extra — impede DELETE em chamado

View

vw_estatisticas_chamados: consolidação de chamados agrupados por categoria, bairro e status usando JOIN LATERAL para buscar o último status.

FAQ de Código e Arquitetura
Perguntas frequentes sobre o funcionamento interno

1. Onde é feita a busca (SELECT) para o login?

Arquivo: backend/portal/views_auth.py (função login_view)

O sistema usa SQL puro via connection.cursor(). A busca é dual: primeiro na tabela Cidadao, depois em Servidor. O SQL puro fica centralizado em backend/portal/db.py.

user = db.buscar_cidadao_por_email(email)
tipo = "cidadao"
if not user:
    user = db.buscar_servidor_por_email(email)
    tipo = "servidor"

2. Como os dados errados são tratados?

Mensagem genérica de erro por segurança: "E-mail ou senha incorretos." — nunca informa se o e-mail existe ou não.

3. O que o sistema memoriza no login?

request.session["usuario_id"] = user.pk
request.session["usuario_tipo"] = tipo
if (user.senha_temporaria or "").strip():
    request.session["forcar_troca_senha"] = True

4. Separação de telas por perfil

Três camadas: (1) Decorators @perfis("GES") nas views, (2) Template com {% if nav_perfil == "GES" %}, (3) Triggers no PostgreSQL via set_config.

5. Criptografia de senhas

Usa make_password() do Django. No banco caem hashes como pbkdf2_sha256$720000$.... A senha real nunca é gravada.

6. Identificação a cada requisição

O PortalUserMiddleware reconstrói o usuário via sessão em cada request, delegando a busca ao db.py.

7. Como as Triggers sabem o perfil?

O middleware executa set_config('portal.perfil', perfil, true) para criar uma sessão PostgreSQL customizada que as triggers consultam.

8. Semáforo do prazo

Função db.popular_status(ch) em backend/portal/db.py. Compara prazo_amarelo_dias e prazo_vermelho_dias com a idade do chamado.

dias = (timezone.now() - ch.dt_abertura).days
if dias >= ch.id_servico.prazo_vermelho_dias:
    ch.cor_semaforo = "vermelho"
elif dias >= ch.id_servico.prazo_amarelo_dias:
    ch.cor_semaforo = "amarelo"
else:
    ch.cor_semaforo = "verde"

9. Burla via DevTools?

Três proteções: decorator @perfis nas views, verificação de permissão na view, e trigger de integridade no banco como última barreira.

10. Triggers do banco

TriggerTabelaO que faz
trg_chamado_after_insert_historico_abchamadoAFTER INSERT — 1º histórico AB
trg_chamado_before_update_metadadoschamadoBEFORE UPDATE — atualiza atualizado_em
trg_historico_after_insert_statushistorico_chamadoAFTER INSERT — dt_conclusao + notificação
trg_foto_chamado_encerradofoto_chamadoBEFORE INSERT — bloqueia se CO/CA
trg_historico_obs_encerradohistorico_chamadoBEFORE INSERT — bloqueia obs se CO/CA
trg_avaliacao_imutavelchamadoBEFORE UPDATE — impede reavaliação

11. Rules do banco

R5: DO INSTEAD NOTHING em UPDATE/DELETE no historico_chamado. Extra: DO INSTEAD NOTHING em DELETE no chamado.

12. View SQL vw_estatisticas_chamados

View que usa JOIN LATERAL para buscar o último status de cada chamado agrupando por categoria, bairro e status. Usada no painel do Gestor.

13. Geração do protocolo

Formato ANO + SEQUENCIAL (ex: 2026000043). Função proximo_protocolo() em utils.py.

14. Notificações automáticas

Trigger 2B insere em notificacao quando um novo histórico é criado. O context_processors.py conta notificações não lidas a cada página.

15. managed = False

Django não gerencia as tabelas. Tudo foi criado manualmente via scripts SQL da pasta database/. Exigência da disciplina de Banco de Dados.

16. transaction.atomic()

Garante atomicidade: se a foto falhar, o chamado não fica sem foto. Ambos os INSERTs funcionam ou nenhum.

17. Índices

6 índices em colunas de chave estrangeira: ix_chamado_cidadao, ix_chamado_bairro, ix_chamado_servico, ix_foto_chamado, ix_historico_chamado, ix_notificacao_chamado.

18. JOIN manual

Como usamos SQL puro, queries com JOIN substituem múltiplos SELECTs. Centralizado em db.py para evitar duplicação.

19. Fluxo de abertura de chamado

5 operações: (1) INSERT chamado, (2) Trigger 1 → histórico AB, (3) Trigger 2B → notificação, (4) INSERT foto, (5) COMMIT. Se falhar, ROLLBACK.

20. Exclusão de chamado pelo Gestor

Usa SQL direto que contorna a Rule: apaga registros filhos (foto, histórico, notificação) antes do chamado pai. Exige justificativa e grava log de auditoria.

Autenticação, Módulo do Cidadão e Página Inicial

Sistema de Autenticação

Autenticação dual: busca primeiro na tabela cidadao, depois em servidor. A sessão guarda usuario_tipo.

  • Redefinição — valida token e atualiza senha_hash
  • Troca obrigatória — 1º acesso de colaborador
