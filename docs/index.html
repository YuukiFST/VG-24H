📋 O que é o VG 24H?

O Portal VG 24H é uma plataforma digital para centralizar serviços públicos do município de Várzea Grande – MT. O objetivo é permitir que cidadãos solicitem serviços (iluminação pública, limpeza, tapa-buracos, poda de árvore etc.), acompanhem o status em tempo real e interajam com a prefeitura de forma transparente.

🔄 Fluxo de um Chamado

1
Cidadão cria chamado

View views_cidadao.py recebe POST — INSERT na tabela chamado. Trigger 1 cria automaticamente o primeiro histórico com status AB (Aberto).

2
Trigger atualiza metadados

Trigger fn_chamado_before_update_metadados atualiza atualizado_em sempre que o chamado é modificado.

3
Equipe visualiza e atribui

View views_equipe.py lista todos os chamados (sem filtro de cidadão). Colaborador atribui a si e muda status.

4
Atualização de status

Novo INSERT em historico_chamado com status EA, EE ou CO. Trigger 2B atualiza dt_conclusao se for CO/CA e gera notificação automática.

5
Conclusão ou Cancelamento

Status CO (Concluído) encerra com sucesso. Status CA (Cancelado) encerra sem conclusão. Cidadão pode avaliar chamados concluídos.

🧩 Conceitos-Chave

O sistema não usa o django.contrib.auth padrão. Possui duas tabelas separadas: cidadao e servidor. O login busca primeiro em cidadão; se não encontrar, busca em servidor.

# views_auth.py — Login Dual
user, tipo = db.buscar_cidadao_por_email(email)
if not user:
    user, tipo = db.buscar_servidor_por_email(email)

if user and check_password(senha, user.senha_hash):
    request.session.cycle_key()
    request.session["usuario_id"] = user.pk
    request.session["usuario_tipo"] = tipo  # 'cidadao' ou 'servidor'
🃏 Pergunta
Por que não usamos o User padrão do Django?
Clique para revelar a resposta
💡 Resposta
Porque o projeto exige dois tipos distintos de usuário com campos diferentes: cidadão (CPF, endereço) e servidor (SIAPE, secretaria, perfil GES/COL). O auth padrão do Django não lida bem com múltiplos perfis sem extensão complexa.

O Django já possui um sistema robusto de hash de senhas via django.contrib.auth.hashers. Usamos make_password() no cadastro e check_password() no login. O algoritmo padrão é PBKDF2 (não bcrypt diretamente).

from django.contrib.auth.hashers import check_password, make_password

# No cadastro (views_auth.py)
senha_hash = make_password(senha_digitada)

# No login (views_auth.py)
if check_password(senha_digitada, user.senha_hash):
    # Login bem-sucedido
🃏 Pergunta
Qual o algoritmo padrão de hash do Django?
Clique para revelar
💡 Resposta
PBKDF2 (Password-Based Key Derivation Function 2) com SHA-256, 600.000 iterações (Django 5.1). O Django pode verificar hashes antigos (bcrypt, SHA1) mas sempre gera novos com o padrão mais seguro.

🧩 Conceitos-Chave

O Django não gerencia as tabelas (não cria nem altera). As tabelas foram criadas manualmente via scripts SQL (database/01_schema.sql). O Django apenas lê e escreve registros usando o ORM para facilitar JOINs e tipagem.

# portal/models.py
class Chamado(models.Model):
    id_chamado = models.AutoField(primary_key=True)
    id_cidadao = models.IntegerField()
    id_servico = models.IntegerField()
    descricao = models.TextField()
    localizacao = models.CharField(max_length=255)
    data_abertura = models.DateTimeField(default=timezone.now)
    atualizado_em = models.DateTimeField(auto_now=True)

    class Meta:
        managed = False   # Django NÃO cria/altera a tabela
        db_table = "chamado"
🃏 Pergunta
Por que usamos managed=False nos models?
Clique para revelar
💡 Resposta
Porque o schema é criado e controlado manualmente via SQL puro (01_schema.sql). O managed=False impede que o Django execute migrations automáticas que poderiam conflitar com nossos triggers, constraints e índices customizados.

Triggers são funções PL/pgSQL que executam automaticamente em resposta a eventos no banco. Temos 3 triggers principais:

  • Trigger 1: Após INSERT em chamado, cria automaticamente um histórico com status AB.
  • Trigger 2A: Antes de UPDATE em chamado, atualiza o campo atualizado_em.
  • Trigger 2B: Após INSERT em historico_chamado, atualiza dt_conclusao e gera notificação.
-- Trigger 1: cria histórico AB automaticamente
CREATE OR REPLACE FUNCTION fn_chamado_after_insert_historico_ab()
RETURNS TRIGGER AS $$
DECLARE
    v_id_status INTEGER;
BEGIN
    SELECT id_status INTO v_id_status
    FROM status_chamado WHERE sigla = 'AB';

    INSERT INTO historico_chamado (id_chamado, id_servidor, id_status, observacao)
    VALUES (NEW.id_chamado, NULL, v_id_status, NULL);

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

Views são consultas salvas que funcionam como tabelas virtuais. A view vw_estatisticas_chamados é usada pelo gestor para relatórios detalhados.

CREATE VIEW vw_estatisticas_chamados AS
SELECT 
    c.id_chamado,
    c.protocolo,
    ci.nome_completo AS cidadao,
    s.nome AS servico,
    st.sigla AS status,
    h.dt_alteracao AS ultima_atualizacao
FROM chamado c
JOIN cidadao ci ON c.id_cidadao = ci.id_cidadao
JOIN servico s ON c.id_servico = s.id_servico
LEFT JOIN historico_chamado h ON c.id_chamado = h.id_chamado
LEFT JOIN status_chamado st ON h.id_status = st.id_status
WHERE h.dt_alteracao = (
    SELECT MAX(dt_alteracao) 
    FROM historico_chamado 
    WHERE id_chamado = c.id_chamado
);
  • 1NF: Cada célula contém apenas um valor (não há listas dentro de células).
  • 2NF: Toda coluna não-chave depende da chave primária inteira (relevante para chaves compostas).
  • 3NF: Não há dependência transitiva: colunas não-chave dependem diretamente da PK, não de outra coluna não-chave.
🃏 Pergunta
Nosso projeto está em qual forma normal?
Clique para revelar
💡 Resposta
3NF. Separamos cidadão, servidor, serviço, categoria, secretaria, bairro e chamado em tabelas distintas. Não há atributos multivalorados (1NF), todas as colunas dependem da PK (2NF), e não há dependências transitivas (3NF).

As views de Equipe e Gestão usam SQL puro via connection.cursor(), não o ORM do Django. Isso permite queries complexas com JOIN LATERAL, subqueries e funções PostgreSQL.

# views_equipe.py — SQL puro
from django.db import connection

with connection.cursor() as cursor:
    cursor.execute("""
        SELECT c.id_chamado, c.protocolo, ci.nome_completo,
               s.nome AS servico, st.sigla AS status
        FROM chamado c
        JOIN cidadao ci ON c.id_cidadao = ci.id_cidadao
        JOIN servico s ON c.id_servico = s.id_servico
        LEFT JOIN LATERAL (
            SELECT id_status, dt_alteracao
            FROM historico_chamado
            WHERE id_chamado = c.id_chamado
            ORDER BY dt_alteracao DESC
            LIMIT 1
        ) h ON true
        LEFT JOIN status_chamado st ON h.id_status = st.id_status
        WHERE st.sigla IN ('AB', 'EA', 'EE')
    """)
    chamados = cursor.fetchall()

FAQ Técnico

Respostas rápidas para as dúvidas mais comuns durante o desenvolvimento.

Porque precisamos de histórico completo de todas as mudanças de status. Se o status fosse um campo simples, perderíamos o rastro de quando mudou de "AB" para "EA" e para "CO". Com a tabela historico_chamado, cada mudança é um registro permanente com data, observação e servidor responsável.

O status atual vem sempre do último registro de historico_chamado para aquele id_chamado.

  • Trigger: Executa uma função PL/pgSQL em resposta a eventos (INSERT, UPDATE, DELETE). Pode ser BEFORE ou AFTER, e row-level ou statement-level. É a forma moderna e recomendada.
  • Rule: Reescreve a query antes de executar. Pode transformar um INSERT em outra operação. Menos flexível que triggers e está em desuso. Nosso projeto removeu as Rules R4 e R6 e migrou o controle de acesso para decorators Django.

Recomendação: Use Triggers para auditoria e integridade. Rules são legado.

  1. Usuário envia email+senha para POST /accounts/login/
  2. View busca em cidadao, depois em servidor (login dual)
  3. Se encontrou, compara senha com check_password() (Django Hashers)
  4. Se válido, salva usuario_id e usuario_tipo no request.session
  5. O middleware PortalUserMiddleware lê a sessão em toda requisição e injeta request.portal_user
  6. Decorators @perfis() verificam o perfil e bloqueiam acesso não autorizado

O PortalUserMiddleware é executado em toda requisição HTTP. Ele faz duas coisas:

  1. Autenticação: Lê o cookie de sessão, busca o usuário no banco e coloca em request.portal_user.
  2. Auditoria: Chama set_config('portal.perfil', ...) e set_config('portal.id_usuario_acao', ...) no PostgreSQL. As triggers do banco leem essas variáveis para saber quem executou a ação e gerar notificações/históricos corretos.

Com managed = False, o Django não cria, altera nem remove as tabelas automaticamente. O schema é controlado manualmente pelos scripts SQL em database/:

  • 01_schema.sql — cria as 11 tabelas com PKs, FKs, CHECKs
  • 02_seed.sql — dados iniciais (status, categorias, serviços)
  • 03_functions_triggers.sql — triggers e funções PL/pgSQL
  • 04_rules.sql — regras de negócio (em desuso, substituídas por decorators)
  • 05_views.sql — views para relatórios

O Django ORM é usado apenas para ler e escrever registros, facilitando JOINs e tipagem em Python.

O semáforo é calculado automaticamente pela property cor_semaforo no model Chamado:

  • Verde: Dias desde a abertura < prazo_amarelo_dias (padrão 15)
  • Amarelo: Dias >= amarelo e < prazo_vermelho_dias (padrão 30)
