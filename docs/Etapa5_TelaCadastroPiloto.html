Etapa 5 - Tela de Cadastro Piloto

Teste das 4 operacoes basicas do BD (CRUD) e recursos avancados

Projeto: VG 24H - Portal de Servicos Publicos de Varzea Grande/MT
Tecnologias: Python/Django + PostgreSQL (NeonDB) + Design System GOV.BR
Entidade piloto: Bairro (cadastro completo com CREATE, READ, UPDATE, DELETE)

Resumo das 4 Operacoes CRUD

Operacao SQL Executado View Django URL
CREATE INSERT INTO bairro (...) VALUES (...) gestao_bairros (POST) /gestao/bairros/
READ SELECT ... FROM bairro WHERE ativo = TRUE gestao_bairros (GET) /gestao/bairros/
UPDATE UPDATE bairro SET nome_bairro = %s, cep = %s ... gestao_bairro_edit /gestao/bairros/<pk>/editar/
DELETE (logico) UPDATE bairro SET ativo = FALSE WHERE ... gestao_bairro_desativar /gestao/bairros/<pk>/desativar/
Nota: O DELETE e logico (soft delete) - o campo ativo e alterado para FALSE em vez de remover o registro do banco. Isso preserva a integridade referencial com a tabela chamado.

1. Schema SQL - Tabela bairro

Definicao da tabela no banco de dados PostgreSQL, com constraints e indice.

SQL database/01_schema.sql 
-- ============================================================
-- Entidade 6: bairro (bairros de Varzea Grande/MT)
-- ============================================================
-- FK apontada por: chamado.id_bairro (bairro onde o chamado foi aberto)
-- ============================================================
CREATE TABLE bairro (
    id_bairro          SERIAL PRIMARY KEY,
    nome_bairro        VARCHAR(100) NOT NULL,
    cep                CHAR(8) NOT NULL,              -- CEP representativo sem mascara (8 digitos)
    regiao             VARCHAR(200),                   -- Ex: "Central", "Norte", "Sul"
    ativo              BOOLEAN NOT NULL DEFAULT TRUE,
    UNIQUE (nome_bairro)
);

-- Indice de performance para JOINs com a tabela chamado
CREATE INDEX ix_chamado_bairro ON chamado (id_bairro);
Constraints aplicadas:
- SERIAL PRIMARY KEY: auto-incremento garantido pelo PostgreSQL
- NOT NULL: nome e CEP sao obrigatorios
- UNIQUE (nome_bairro): nao permite bairros com nome duplicado
- DEFAULT TRUE: todo bairro nasce ativo

3. Model Django - Classe Bairro

Mapeamento ORM da tabela bairro. Usa managed = False (tabela criada via SQL, nao pelo Django).

PYTHON backend/portal/models.py 
class Bairro(models.Model):
    """
    Mapeamento da tabela bairro.
    Bairros de Varzea Grande/MT. FK apontada por: Chamado.id_bairro.
    """
    id_bairro = models.AutoField(primary_key=True)
    nome_bairro = models.CharField(max_length=100, unique=True)
    cep = models.CharField(max_length=8)
    regiao = models.CharField(max_length=200, blank=True, null=True)
    ativo = models.BooleanField(default=True)

    class Meta:
        managed = False          # Django NAO gerencia (tabela criada via SQL)
        db_table = "bairro"

    def __str__(self):
        return self.nome_bairro

6. Controle de Acesso - Decorator @perfis

Todas as views de bairro usam @perfis("GES") - apenas gestores podem acessar.

PYTHON backend/portal/decorators.py 
def perfis(*codigos):
    """
    Decorator que restringe acesso por perfil.
    Exemplo: @perfis("GES") -> apenas Gestores
             @perfis("COL", "GES") -> Colaboradores e Gestores
    """
    def dec(view):
        @wraps(view)
        def w(request, *args, **kwargs):
            u = request.portal_user
            if not u:
                return redirect("portal:login")         # Nao logado -> login
            if perfil_codigo(u) not in codigos:
                messages.error(request, "Sem permissao para acessar esta area.")
                return redirect("portal:root")           # Perfil errado -> home
            return view(request, *args, **kwargs)        # Autorizado -> executa
        return w
    return dec

9. Operacao UPDATE - Editar Bairro

O gestor clica em "Editar" na tabela. A view busca o bairro por ID (SELECT), mostra o formulario preenchido, e ao enviar executa um UPDATE.

PYTHON backend/portal/views_gestao.py 
@perfis("GES")
@require_http_methods(["GET", "POST"])
def gestao_bairro_edit(request, pk):
    # SQL puro: SELECT bairro por ID
    with connection.cursor() as cursor:
        cursor.execute(
            "SELECT id_bairro, nome_bairro, cep, regiao, ativo "
            "FROM bairro WHERE id_bairro = %s",
            [pk],
        )
        row = cursor.fetchone()
    if not row:
        raise Http404()

    obj = Bairro()
    obj.id_bairro = row[0]
    obj.nome_bairro = row[1]
    obj.cep = row[2]
    obj.regiao = row[3]
    obj.ativo = row[4]
    obj._state.adding = False

    if request.method == "POST":
        form = BairroForm(request.POST, instance=obj)
        if form.is_valid():
            d = form.cleaned_data
            # SQL puro: UPDATE bairro
            with connection.cursor() as cursor:
                cursor.execute(
                    "UPDATE bairro SET nome_bairro = %s, cep = %s, regiao = %s "
                    "WHERE id_bairro = %s",
                    [d["nome_bairro"], d["cep"], d.get("regiao"), pk],
                )
            messages.success(request, "Bairro atualizado.")
            return redirect("portal:gestao_bairros")
    else:
        form = BairroForm(instance=obj)
    return render(
        request,
        "portal/gestao/bairro_form.html",
        {"form": form, "obj": obj},
    )
SQL executado (leitura):
SELECT id_bairro, nome_bairro, cep, regiao, ativo FROM bairro WHERE id_bairro = 5

SQL executado (gravacao):
UPDATE bairro SET nome_bairro = 'Novo Nome', cep = '78110100', regiao = 'Central' WHERE id_bairro = 5
PRINT 4 Formulario de edicao do bairro preenchido /gestao/bairros/<pk>/editar/ Clique no botao de editar (icone de lapis) em qualquer bairro da tabela. A tela mostra o formulario com os campos ja preenchidos com os dados atuais do bairro (nome, checkbox "Ativo"). Tire o print com os dados visiveis.

10. Operacao DELETE (Logico) - Desativar Bairro

O gestor clica no botao "Desativar" e confirma. A view executa um UPDATE que muda o campo ativo para FALSE.

PYTHON backend/portal/views_gestao.py 
@perfis("GES")
@require_http_methods(["POST"])
def gestao_bairro_desativar(request, pk):
    # SQL puro: verifica existencia e desativa bairro
    with connection.cursor() as cursor:
        cursor.execute("SELECT id_bairro FROM bairro WHERE id_bairro = %s", [pk])
        if not cursor.fetchone():
            raise Http404()
        cursor.execute(
            "UPDATE bairro SET ativo = FALSE WHERE id_bairro = %s", [pk]
        )
    messages.info(request, "Bairro inativado.")
    return redirect("portal:gestao_bairros")
SQL executado:
UPDATE bairro SET ativo = FALSE WHERE id_bairro = 5

Por que soft delete? A tabela chamado tem uma FK para bairro. Se deletasse fisicamente, chamados antigos ficariam orfaos. O soft delete preserva o historico enquanto impede novos usos.
PRINT 5 Mensagem "Bairro inativado" apos desativar /gestao/bairros/ Clique no botao de desativar (icone de ban) em algum bairro e confirme na caixa de dialogo. A pagina recarrega com a mensagem "Bairro inativado." no topo. O bairro some da lista (filtro ativo = TRUE). Tire o print mostrando a mensagem + a tabela atualizada.

13. Recurso Avancado de BD: Triggers

O sistema possui 3 triggers PL/pgSQL que automatizam acoes criticas no banco de dados.

Trigger 1 - Auto-criar historico "Aberto" ao inserir chamado

Quando um chamado e inserido, o trigger cria automaticamente o primeiro registro de historico com status "AB" (Aberto).

SQL (PL/pgSQL) database/03_functions_triggers.sql 
CREATE OR REPLACE FUNCTION fn_chamado_after_insert_historico_ab()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
DECLARE
    v_id_status INTEGER;
BEGIN
    -- Busca o ID do status 'AB' na tabela status_chamado
    SELECT id_status INTO v_id_status
    FROM status_chamado WHERE sigla = 'AB';

    -- Insere o primeiro historico: status AB, servidor = NULL (sistema)
    INSERT INTO historico_chamado (id_chamado, id_servidor, id_status, observacao)
    VALUES (NEW.id_chamado, NULL, v_id_status, NULL);

    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_chamado_after_insert_historico_ab
    AFTER INSERT ON chamado
    FOR EACH ROW
    EXECUTE PROCEDURE fn_chamado_after_insert_historico_ab();

Trigger 2A - Auto-atualizar timestamp em UPDATE

Atualiza automaticamente o campo atualizado_em sempre que qualquer UPDATE ocorre na tabela chamado.

SQL (PL/pgSQL) database/03_functions_triggers.sql 
CREATE OR REPLACE FUNCTION fn_chamado_before_update_metadados()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
BEGIN
    -- BEFORE UPDATE: modifica NEW antes da gravacao
    NEW.atualizado_em := CURRENT_TIMESTAMP;
    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_chamado_before_update_metadados
    BEFORE UPDATE ON chamado
    FOR EACH ROW
    EXECUTE PROCEDURE fn_chamado_before_update_metadados();

Trigger 2B - Auto-notificar e atualizar conclusao ao mudar status

Ao inserir um novo historico (mudanca de status), o trigger: (1) atualiza dt_conclusao se for CO/CA, (2) gera notificacao automatica para o cidadao.

SQL (PL/pgSQL) database/03_functions_triggers.sql 
CREATE OR REPLACE FUNCTION fn_historico_after_insert_status()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
DECLARE
    nova_sigla  CHAR(2);
    desc_nova   VARCHAR(200);
    v_protocolo VARCHAR(20);
    msg         VARCHAR(200);
BEGIN
    -- Busca a sigla e descricao do status inserido
    SELECT sigla, descricao INTO nova_sigla, desc_nova
    FROM status_chamado WHERE id_status = NEW.id_status;

    -- Busca o numero de protocolo do chamado
    SELECT num_protocolo INTO v_protocolo
    FROM chamado WHERE id_chamado = NEW.id_chamado;

    -- CO ou CA: marca data de conclusao
    IF nova_sigla IN ('CO', 'CA') THEN
        UPDATE chamado SET dt_conclusao = COALESCE(dt_conclusao, CURRENT_TIMESTAMP)
        WHERE id_chamado = NEW.id_chamado;
    -- AB, EA ou EE: limpa dt_conclusao (reaberto)
    ELSIF nova_sigla IN ('AB', 'EA', 'EE') THEN
        UPDATE chamado SET dt_conclusao = NULL
        WHERE id_chamado = NEW.id_chamado;
    END IF;

    -- Gera notificacao automatica
    msg := left(
        'Chamado ' || v_protocolo || ': status alterado para '
        || COALESCE(desc_nova, nova_sigla),
        200
    );
    INSERT INTO notificacao (id_chamado, mensagem)
    VALUES (NEW.id_chamado, msg);

    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_historico_after_insert_status
    AFTER INSERT ON historico_chamado
    FOR EACH ROW
    EXECUTE PROCEDURE fn_historico_after_insert_status();

14. Recurso Avancado de BD: Rules de Integridade

Rules e triggers que garantem regras de negocio diretamente no banco, independente da aplicacao.

R1 - Bloquear fotos em chamados encerrados

SQL (PL/pgSQL) database/04_rules.sql 
CREATE OR REPLACE FUNCTION fn_rule_foto_chamado_encerrado()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
    IF EXISTS (
        SELECT 1
        FROM historico_chamado hc
        JOIN status_chamado s ON s.id_status = hc.id_status
        WHERE hc.id_chamado = NEW.id_chamado
          AND s.sigla IN ('CO', 'CA')
          AND hc.dt_alteracao = (
              SELECT MAX(dt_alteracao) FROM historico_chamado
              WHERE id_chamado = NEW.id_chamado
          )
    ) THEN
        RAISE EXCEPTION 'Chamado encerrado (CO/CA): nao e permitido adicionar fotos.';
    END IF;
    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_foto_chamado_encerrado
    BEFORE INSERT ON foto_chamado

Diferenca entre Rule e Trigger:
- Rule: reescreve a query ANTES de executar (ex: transforma DELETE em "nada")
- Trigger: executa codigo PL/pgSQL ANTES ou DEPOIS da operacao (pode validar e lancar excecao)

15. Recurso Avancado de BD: View SQL com JOIN LATERAL

View que calcula estatisticas de chamados agrupados por categoria, bairro e status, usando JOIN LATERAL para buscar o ultimo status de cada chamado.

SQL database/05_views.sql 
CREATE VIEW vw_estatisticas_chamados AS
SELECT
    cat.nome        AS categoria,          -- Nome da categoria de servico
    b.nome_bairro   AS bairro,             -- Nome do bairro
    s.sigla         AS sigla_status,       -- Sigla do status (AB, EA, EE, CO, CA)
    s.descricao     AS status_descricao,   -- Descricao do status
    COUNT(*)::BIGINT AS total_chamados     -- Quantidade de chamados
FROM chamado ch
-- JOIN 1: chamado -> servico -> categoria_servico
JOIN servico srv     ON srv.id_servico  = ch.id_servico
JOIN categoria_servico cat ON cat.id_categoria = srv.id_categoria
-- JOIN 2: chamado -> bairro
JOIN bairro b        ON b.id_bairro     = ch.id_bairro
-- JOIN 3: LATERAL - busca o ULTIMO status de cada chamado
-- A subconsulta roda para CADA chamado, ordena por dt_alteracao DESC
-- e pega apenas 1 registro (o mais recente)
JOIN LATERAL (
    SELECT id_status FROM historico_chamado
    WHERE id_chamado = ch.id_chamado
    ORDER BY dt_alteracao DESC LIMIT 1
) ultimo_h ON TRUE
-- JOIN 4: status_chamado - sigla e descricao
JOIN status_chamado s ON s.id_status = ultimo_h.id_status
-- Filtra apenas registros ativos
WHERE cat.ativo AND srv.ativo AND b.ativo
-- Agrupamento: um registro por (categoria, bairro, status)
GROUP BY cat.nome, b.nome_bairro, s.sigla, s.descricao;
O que e JOIN LATERAL?
Diferente de um JOIN comum, o JOIN LATERAL permite que a subconsulta referencie colunas da tabela anterior (neste caso, ch.id_chamado). Funciona como um "foreach" no SQL: para cada chamado, busca o ultimo historico. Sem LATERAL, seria necessario uma subconsulta mais complexa com ROW_NUMBER() ou DISTINCT ON.

16. Recurso Avancado de BD: Variaveis de Sessao PostgreSQL

O middleware Django injeta variaveis de sessao no PostgreSQL usando set_config(). As triggers e functions do banco acessam essas variaveis para saber qual usuario/perfil esta executando a acao (auditoria).

PYTHON backend/portal/middleware.py 
def _postgres_sessao(perfil, id_acao):
    """
    Injeta variaveis de sessao no PostgreSQL.
    As triggers acessam via: SELECT current_setting('portal.perfil', true);
    """
    perfil = perfil or ""
    id_acao = "" if id_acao is None else str(id_acao)
    with connection.cursor() as c:
        c.execute("SELECT set_config('portal.perfil', %s, true)", [perfil])
        c.execute("SELECT set_config('portal.id_usuario_acao', %s, true)", [id_acao])


class PortalUserMiddleware:
    """
    Executado em TODA requisicao ANTES de chegar na view.
    Efeitos:
      - request.portal_user -> objeto do usuario logado (ou None)
      - set_config() no PG -> triggers sabem quem esta agindo
    """
    def __init__(self, get_response):
        self.get_response = get_response

    def __call__(self, request):
        user = _usuario_da_sessao(request)
        request.portal_user = user

        # Informa ao PostgreSQL quem esta operando
        if user:
            _postgres_sessao((user.perfil or "").strip(), user.pk)
        else:
            _postgres_sessao(None, None)

        return self.get_response(request)
Fluxo completo:
1. Usuario faz login -> sessao Django armazena usuario_id e usuario_tipo
2. Toda requisicao -> middleware busca o usuario no banco via SQL puro
3. Middleware chama set_config('portal.perfil', 'GES', true) no PostgreSQL
4. Triggers podem ler o perfil via current_setting('portal.perfil', true) para auditoria
