Importante (não erre isso): o projeto usa SQL puro (comando cursor.execute(...)), não o ORM do Django, porque a professora exigiu SQL puro. Isso é um requisito da disciplina — não apresente como um mérito/escolha do grupo. Os models existem só para formulários e propriedades; todas as leituras e escritas de dados são SQL escrito à mão.
Onde estão os arquivos: back-end e SQL em backend/portal/ e database/; telas (HTML) em backend/templates/portal/.

Arquitetura em 1 minuto (todos devem saber)

  • SQL puro via o pacote db/: as views chamam funções do pacote backend/portal/db/ — dividido por domínio (cidadao, servidor, chamado, historico, catalogo, stats, notificacao, senha, config, pagination, _shared) — usando with connection.cursor() as cursor: cursor.execute(sql, params). O pacote é importado como from portal import db e as funções são chamadas por db.<funcao>().
  • Status por "event sourcing": a tabela chamado NÃO tem coluna de status. O status atual é o último registro de historico_chamado (ORDER BY dt_alteracao DESC LIMIT 1), normalmente via JOIN LATERAL.
  • Triggers no banco fazem regras automáticas: ao abrir chamado cria o histórico "AB"; ao mudar status seta/limpa dt_conclusao e gera a notificação.
  • Semáforo (verde/amarelo/vermelho) = dias em aberto comparados aos prazos globais (configuracao_semaforo.prazo_amarelo_dias / prazo_vermelho_dias).
  • Perfis: CID (cidadão), COL (colaborador), GES (gestor).

DER: Conceitual × Lógico (como explicar)

  • Conceitual: a visão de alto nível — as entidades e como se relacionam, sem detalhe de tipo. Ex.: um Cidadão abre vários Chamados; cada Chamado pertence a um Serviço, a um Bairro e tem vários registros de Histórico; cada Serviço pertence a uma Categoria.
  • Lógico: o conceitual já traduzido em tabelas, colunas, PK/FK e tipos — é o database/01_schema.sql (mostre este arquivo). Cada relação vira uma chave estrangeira (REFERENCES); cada regra de valor vira um CHECK.
  • 11 entidades do DER + 3 tabelas de apoio (configuracao_semaforo, protocolo_seq, banner_publicacao) = 14 tabelas no schema.
  • Detalhe forte para citar: o status do chamado não é coluna — é o último registro de historico_chamado (padrão event sourcing).

Visão geral — como o SQL é executado base
"Centralizamos as consultas no pacote db/, dividido por domínio (cidadao.py, chamado.py, catalogo.py, stats.py, etc.). Cada função abre um cursor, executa um comando SQL com parâmetros (%s, que evita SQL injection) e transforma as linhas em objetos. As views importam from portal import db e chamam db.<funcao>() — não precisam saber em qual módulo a função está."
O pacote backend/portal/db/ tem um __init__.py (fachada) que re-exporta todas as funções públicas. Por isso o nome de chamada continua db.buscar_cidadao_por_email(...), mesmo a função morando em db/cidadao.py. Leitores "planos" (listas simples) usam um pequeno helper fetch_all/fetch_one em db/_shared.py que executa o SELECT e mapeia as linhas.

            "FROM cidadao "

            [email],

        row = cursor.fetchone()
  • with connection.cursor() as cursor abre a conexão com o PostgreSQL.
  • cursor.execute(sql, [email]) roda o comando; o %s é substituído pelo valor de forma segura (parâmetro, não concatenação de texto).
  • LOWER(email) = %s faz a busca sem diferenciar maiúsculas/minúsculas.
  • ativo = TRUE: só considera contas ativas (o sistema usa "soft delete").
  • fetchone() traz 1 linha; fetchall() traria todas.

Histórico — a "fonte da verdade" do status

CREATE TABLE historico_chamado (
    id_historico_chamado   SERIAL PRIMARY KEY,
    dt_alteracao           TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    observacao             VARCHAR(500) CHECK (observacao IS NULL OR char_length(observacao) <= 500),
    id_chamado             INTEGER NOT NULL REFERENCES chamado (id_chamado),
    id_servidor            INTEGER REFERENCES servidor (id_servidor),
    id_status              INTEGER NOT NULL REFERENCES status_chamado (id_status)
);

CREATE INDEX ix_historico_chamado_status ON historico_chamado (id_chamado, dt_alteracao);
  • Cada mudança de status é um novo INSERT aqui (nunca um UPDATE). Por isso fica o histórico completo.
  • id_servidor pode ser NULL (ex.: quando o sistema cria o status automaticamente).
  • O índice em (id_chamado, dt_alteracao) acelera a busca do "último status".

Abrir chamado — protocolo + INSERT + Trigger 1 INSERT
"Para abrir um chamado eu gero o número de protocolo de forma atômica (sem dois iguais), faço o INSERT na tabela chamado e registro a foto. Eu não insiro o status inicial — um trigger do banco cria sozinho o histórico 'AB' (Aberto)."

database/03_functions_triggers.sql — fn_proximo_protocolo (linhas ~148-176)
CREATE OR REPLACE FUNCTION fn_proximo_protocolo(p_ano INTEGER)
RETURNS VARCHAR LANGUAGE plpgsql AS $$
DECLARE v_numero INTEGER; v_protocolo VARCHAR(20);
BEGIN
    LOOP
        -- passo ATÔMICO: cria a linha do ano com 1, ou incrementa +1 se já existe
        INSERT INTO protocolo_seq (ano, ultimo_numero) VALUES (p_ano, 1)
        ON CONFLICT (ano) DO UPDATE SET ultimo_numero = protocolo_seq.ultimo_numero + 1
        RETURNING ultimo_numero INTO v_numero;
        -- monta ANO + 6 dígitos com zeros à esquerda
        v_protocolo := p_ano::TEXT || lpad(v_numero::TEXT, 6, '0');
        -- se esse protocolo ainda não existe em chamado, está livre -> sai do laço
        EXIT WHEN NOT EXISTS (SELECT 1 FROM chamado WHERE num_protocolo = v_protocolo);
    END LOOP;
    RETURN v_protocolo;
END; $$;
  • Stored function = função salva dentro do banco (regra de negócio no PostgreSQL). É um recurso SQL avançado que a professora costuma pedir — mostre este arquivo.
  • INSERT ... ON CONFLICT (ano) DO UPDATE ... RETURNING: é um "UPSERT" atômico. Se o ano não existe, cria com 1; se já existe, soma +1 — sem risco de dois chamados pegarem o mesmo número.
  • LOOP ... EXIT WHEN NOT EXISTS(...): se por dessincronização o número já estiver ocupado, avança sozinho até achar um livre.
  • A aplicação (utils.proximo_protocolo) só faz SELECT fn_proximo_protocolo(ano).

Status atual — o truque do JOIN LATERAL consulta-chave
"Como o chamado não tem status, em toda listagem eu preciso descobrir o último status de cada chamado. FROM chamado c
JOIN servico s ON c.id_servico = s.id_servico
JOIN bairro b ON c.id_bairro = b.id_bairro
JOIN LATERAL (

Semáforo e prioridade (verde/amarelo/vermelho) CASE / agregação
"O semáforo compara quantos dias o chamado está aberto com os prazos globais (tabela configuracao_semaforo). Tem duas formas no código: uma função simples por chamado, e uma consulta SQL que conta tudo de uma vez (verde/amarelo/vermelho) usando CASE WHEN e intervalos de dias."
backend/portal/db/stats.py — cor_semaforo (~linha 24)
def cor_semaforo(dt_abertura, prazo_amarelo_dias, prazo_vermelho_dias):
    dias = (timezone.now() - dt_abertura).days
    if dias >= prazo_vermelho_dias:
        return "vermelho"
    if dias >= prazo_amarelo_dias:
        return "amarelo"
    return "verde"
  • dias = (agora - dt_abertura).days: dias desde a abertura.
  • Se passou do prazo vermelho → "vermelho"; senão se passou do amarelo → "amarelo"; senão "verde".
backend/portal/db/stats.py — calcular_stats_semaforo (~linha 41)
SELECT
  COALESCE(SUM(CASE WHEN (%s - c.dt_abertura) < make_interval(days := cfg.prazo_amarelo_dias) THEN 1 ELSE 0 END), 0),
  COALESCE(SUM(CASE WHEN (%s - c.dt_abertura) >= make_interval(days := cfg.prazo_amarelo_dias)
    AND (%s - c.dt_abertura) < make_interval(days := cfg.prazo_vermelho_dias) THEN 1 ELSE 0 END), 0),
  COALESCE(SUM(CASE WHEN (%s - c.dt_abertura) >= make_interval(days := cfg.prazo_vermelho_dias) THEN 1 ELSE 0 END), 0)
FROM chamado c
CROSS JOIN configuracao_semaforo cfg
WHERE TRUE
  • CROSS JOIN configuracao_semaforo cfg: junta a única linha de prazos globais a cada chamado.
  • make_interval(days := cfg.prazo_amarelo_dias): transforma o número de dias num intervalo de tempo para comparar com (agora - dt_abertura).
  • Cada SUM(CASE WHEN ... THEN 1 ELSE 0 END) conta uma faixa: no prazo (verde), atenção (amarelo) e crítico (vermelho), tudo em uma só consulta.
  • COALESCE(..., 0): se não houver chamados, devolve 0 em vez de NULL.
  • Há ainda calcular_stats_semaforo_por_servico(), que faz a mesma contagem agrupada por serviço (GROUP BY s.id_servico, s.nome).
  • A prioridade (0 a 5) é um campo separado em chamado.prioridade, definido pela equipe na tela de edição (ver card "Editar chamado").

Trigger 2B — conclusão + notificação (linhas ~79-126)

CREATE OR REPLACE FUNCTION fn_historico_after_insert_status()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
DECLARE nova_sigla CHAR(2); desc_nova VARCHAR(200); v_protocolo VARCHAR(20); msg VARCHAR(200);
BEGIN
    SELECT sigla, descricao INTO nova_sigla, desc_nova
    FROM status_chamado WHERE id_status = NEW.id_status;
    SELECT num_protocolo INTO v_protocolo FROM chamado WHERE id_chamado = NEW.id_chamado;

    IF nova_sigla IN ('CO', 'CA') THEN
        UPDATE chamado SET dt_conclusao = COALESCE(dt_conclusao, CURRENT_TIMESTAMP)
        WHERE id_chamado = NEW.id_chamado;
    ELSIF nova_sigla IN ('AB', 'EA', 'EE') THEN
        UPDATE chamado SET dt_conclusao = NULL WHERE id_chamado = NEW.id_chamado;
    END IF;

    msg := left('Chamado ' || v_protocolo || ': status alterado para ' || COALESCE(desc_nova, nova_sigla), 200);
    INSERT INTO notificacao (id_chamado, mensagem) VALUES (NEW.id_chamado, msg);
    RETURN NEW;
END; $$;

CREATE TRIGGER trg_historico_after_insert_status
    AFTER INSERT ON historico_chamado FOR EACH ROW
    EXECUTE PROCEDURE fn_historico_after_insert_status();
  • Dispara AFTER INSERT ON historico_chamado — ou seja, sempre que muda o status.
  • Se o novo status é CO/CA, grava a data de conclusão; se for AB/EA/EE (reaberto), limpa a data.
  • Monta a mensagem e cria a notificação. Centraliza a regra no banco — a aplicação só insere o status.
  • O Trigger 2A (BEFORE UPDATE ON chamado) só faz NEW.atualizado_em := CURRENT_TIMESTAMP.

CREATE OR REPLACE FUNCTION fn_rule_avaliacao_imutavel()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
    IF OLD.nota_avaliacao IS NOT NULL
       AND (NEW.nota_avaliacao IS DISTINCT FROM OLD.nota_avaliacao
            OR NEW.comentario_avaliacao IS DISTINCT FROM OLD.comentario_avaliacao)
    THEN
        RAISE EXCEPTION 'A avaliacao ja foi registrada e nao pode ser alterada.';
    END IF;
    RETURN NEW;
END; $$;

  • IS DISTINCT FROM compara tratando NULL como valor (evita problema do = NULL).
  • Se já havia nota e alguém tenta mudar, RAISE EXCEPTION bloqueia.

def perfis(*codigos):
    def dec(view):
        @wraps(view)
        def w(request, *args, **kwargs):
            u = request.portal_user
            if not u:
                return redirect("portal:login")
            if perfil_codigo(u) not in codigos:
                messages.error(request, "Sem permissão para acessar esta área.")
                return redirect("portal:root")
            return view(request, *args, **kwargs)
        return w
    return dec

Respostas rápidas (perguntas que a professora costuma fazer)

Use isto como "cola" final. Cada pergunta é um card — clique para abrir a resposta. Cada resposta diz o arquivo e quem responde.

"Onde o código verifica se o login está certo ou errado?"
Bruno: em backend/portal/views_auth.py, função login_view (~linha 115): check_password(senha, user.senha_hash) compara a senha digitada com o hash do banco. Antes disso, o SELECT em db.buscar_cidadao_por_email / buscar_servidor_por_email (Rafael) decide se o e-mail existe.
"Onde aparece a mensagem dizendo que o login está certo/errado?"
Bruno: a mensagem é definida na view com messages.error(request, "E-mail ou senha incorretos.") (erro) e messages.success(request, f"Olá, ...") (sucesso), em views_auth.py. Yuuki: ela é exibida no template base.html (bloco {% if messages %}, ~linha 207), virando uma caixa br-message danger (vermelha) ou success (verde).
"Onde está a prioridade do chamado?"
Rafael: coluna chamado.prioridade INTEGER CHECK (>=0 AND <=5) (database/01_schema.sql). Bruno: é gravada no UPDATE chamado SET prioridade=%s dentro de alterar_status (services/chamado.py), chamado pela view equipe_chamado_status. Yuuki: aparece como tag colorida na tabela (equipe/dashboard.html, ~linha 236).
"Onde está o semáforo (verde/amarelo/vermelho)?"
Rafael: em db/stats.py: função cor_semaforo (~linha 24) e a consulta calcular_stats_semaforo (~linha 41) com CASE WHEN ... make_interval(days := cfg....). Os prazos são globais, na tabela configuracao_semaforo (prazo_amarelo_dias / prazo_vermelho_dias), trazida por CROSS JOIN. Yuuki: os 4 cards e a legenda em equipe/dashboard.html (~linha 47).
"Onde o código mostra o dashboard?"
Bruno: dois pontos — equipe_dashboard (gráficos, rota /equipe/) e equipe_chamados_lista (lista + semáforo, rota /equipe/chamados/), em views_equipe.py. Yuuki: templates equipe/estatisticas_graficos.html e equipe/dashboard.html. Rafael: as consultas de contagem (db.buscar_stats_dashboard) e a view vw_estatisticas_chamados.
"Em quantos dias o chamado está aberto — onde isso é calculado?"
Bruno/Rafael: na função formatar_dias_em_aberto(dt_abertura) (utils.py, ~linha 76), que devolve um texto como "12 dia(s)" (ou minutos/horas se for recente). É usada ao montar a lista da equipe (db.listar_chamados_equipe) e no detalhe. Yuuki: exibido com {{ linha.dias_aberto }} numa tag colorida pela cor do semáforo.
"Onde a notificação é criada quando o usuário recebe uma?"
Rafael: no Trigger 2B fn_historico_after_insert_status (database/03_functions_triggers.sql, ~linha 79): a cada novo status inserido, ele faz INSERT INTO notificacao (...). O código Python não cria notificação — só lê/apaga (cidadao_notificacoes).
"Como vocês sabem o status atual se o chamado não tem coluna de status?"
Rafael: o status é o último registro de historico_chamado (ORDER BY dt_alteracao DESC LIMIT 1), buscado com JOIN LATERAL nas listagens e por db.popular_status no detalhe. Toda mudança é um INSERT novo — nunca UPDATE — então fica o histórico completo.
"Como o número de protocolo é gerado sem repetir?"
Rafael: a stored function fn_proximo_protocolo(ano) no banco, com INSERT INTO protocolo_seq ... ON CONFLICT (ano) DO UPDATE SET ultimo_numero = ... + 1 RETURNING dentro de um LOOP — operação atômica. A app só faz SELECT fn_proximo_protocolo(ano) (utils.proximo_protocolo). Formato: ano + 6 dígitos (ex.: 2026000001).
"Quais CRUDs o sistema tem?"
Chamados (abrir/editar/excluir), Banners, Bairros, Serviços, Categorias, Colaboradores e os prazos globais do semáforo, além de Cadastro/Login, troca/recuperação de senha e Notificações. Todos têm card próprio em cada integrante.
"Por que SQL puro e não o ORM do Django?"
Porque foi requisito da disciplina de Banco de Dados (SQL puro, sem ORM para dados). Os models existem só para formulários e propriedades; toda leitura/escrita é cursor.execute() no pacote db/, nas services e em algumas views.
"Por que guardar o status no histórico em vez de uma coluna? Qual a vantagem?"
Rafael: é o padrão event sourcing. Cada mudança vira um INSERT em historico_chamado, então fica registrado quem mudou, quando e para qual status — rastreabilidade/auditoria total, e o histórico é imutável. O custo é precisar do JOIN LATERAL para descobrir o status atual em cada consulta.
"O que é o JOIN LATERAL e por que usar em vez de uma subconsulta comum?"
Rafael: o JOIN LATERAL roda uma subconsulta para cada chamado, enxergando o c.id_chamado da linha de fora. Assim pego o último histórico (ORDER BY dt_alteracao DESC LIMIT 1) de todos os chamados numa única passada, trazendo várias colunas do status ao mesmo tempo e evitando o problema N+1. Fica no helper sql_lateral_ultimo_status().
"Como o sistema evita SQL injection?"
Rafael: todo valor entra por parâmetro %s (nunca concatenando texto da entrada do usuário). Onde é preciso interpolar nome de tabela/coluna (ex.: o helper do JOIN LATERAL), o código só aceita itens de uma lista branca (COLUNAS_LATERAL_VALIDAS, _validar_tabela) e valida o alias.
"Como as senhas são guardadas e verificadas?"
Bruno: nunca em texto puro. Ao gravar, make_password gera um hash bcrypt (coluna senha_hash); ao logar, check_password(senha, user.senha_hash) compara sem desfazer o hash. Vale para cidadãos e servidores.
"O que impede um cidadão de ver ou mexer no chamado de outro?"
Bruno: a função _chamado_do_cidadao levanta Http404 se o chamado não for do usuário logado; as listagens e notificações filtram por WHERE c.id_cidadao = %s. Devolve 404 (não 403) de propósito, para não revelar que o recurso existe.
"O histórico pode ser alterado? Como vocês garantem a auditoria?"
Rafael: triggers em database/04_rules.sql bloqueiam qualquer UPDATE em historico_chamado e só liberam DELETE no fluxo oficial de exclusão (a aplicação seta set_config('portal.excluindo','true')). A avaliação também fica imutável depois de registrada (RAISE EXCEPTION).
"O que acontece no banco quando o status muda?"
Rafael: o INSERT em historico_chamado dispara dois gatilhos: o Trigger 2A atualiza chamado.atualizado_em, e o Trigger 2B seta/limpa dt_conclusao (CO/CA preenche; AB/EA/EE limpa) e cria a notificação para o cidadão. A aplicação só insere o status — o resto é automático.
"Qual a diferença entre Colaborador (COL) e Gestor (GES)?"
Bruno: os dois veem e atendem chamados, mas só o GES pode excluir chamado e mexer em chamado já encerrado (CO/CA); o COL é bloqueado nesses casos (no Python). E só o GES vê os menus de Serviços, Colaboradores e Banners. O perfil vem da coluna servidor.perfil.
"Como funciona o primeiro acesso de um colaborador?"
Bruno: o gestor cria o colaborador com uma senha provisória e senha_temporaria = '1'. No login, isso redireciona para troca_senha_obrigatoria e ele só usa o sistema depois de trocar a senha.
"Tem recuperação de senha por e-mail?"
Bruno: sim. recuperar_senha_view envia um e-mail com um token assinado (signing.dumps, salt vg.pwreset, validade de 3 dias). redefinir_senha_view valida o token e grava a nova senha (em hash). Por segurança, a mensagem é sempre "se o e-mail existir...".
"Como vocês limitam tentativas de login (força bruta)?"
Bruno: _check_login_rate_limit guarda as tentativas na sessão e bloqueia após 5 tentativas em 5 minutos (MAX_LOGIN_ATTEMPTS = 5, LOGIN_TIMEOUT = 300), mostrando "Muitas tentativas. Aguarde 5 minutos.".
"O que é \"soft delete\" e onde é usado?"
Rafael: em vez de apagar, marca ativo = FALSE (bairro, serviço, categoria, colaborador, secretaria, cidadão). Assim o registro some das listas mas o histórico continua válido. O único CRUD com DELETE físico é o de Banners.
"Por que usar ModelForm do Django se vocês gravam com SQL puro?"
Bruno: o ModelForm é usado só para validar (campos obrigatórios, formato de e-mail, etc.). Nunca chamamos form.save() — depois de validar, a gravação é feita por cursor.execute() (SQL puro), atendendo ao requisito da disciplina.
"Como o middleware participa da segurança no banco?"
Bruno: o PortalUserMiddleware roda a cada requisição: carrega o usuário em request.portal_user e injeta set_config('portal.perfil', ...) na sessão do PostgreSQL. Esses valores são usados pelos triggers de regra/auditoria no banco.
"Por que os prazos do semáforo ficam numa tabela separada?"
Rafael: configuracao_semaforo é um singleton (CHECK (id = 1)): uma única linha com os prazos globais. Assim o gestor ajusta os prazos numa só tela e eles valem para todos os serviços — não é preciso repetir prazo em cada serviço.
"Por que o acesso a dados é um pacote db/ e não um único arquivo db.py?"
Rafael/Bruno: a camada de SQL puro foi organizada num pacote backend/portal/db/ dividido por domínio (cidadao, servidor, chamado, historico, catalogo, stats, notificacao, senha, config, pagination e _shared). Um __init__.py (fachada) re-exporta todas as funções públicas, então quem chama continua usando from portal import db e db.<funcao>() sem precisar saber o módulo. Vantagem: cada arquivo fica pequeno e coeso, em vez de um arquivo gigante com tudo misturado.
"Se eu digitar um e-mail que não existe no login, o que acontece e onde?"
Bruno: os SELECT db.buscar_cidadao_por_email e buscar_servidor_por_email não retornam linha, então user is None e a view chama _record_login_attempt + messages.error("E-mail ou senha incorretos.") (views_auth.login_view). De propósito, a mensagem é a mesma de "senha errada".
"Por que a mensagem de erro de login é a mesma para e-mail inexistente e senha errada?"
Bruno: por segurança (não revelar se um e-mail está cadastrado). Os dois ramos da view usam exatamente messages.error(request, "E-mail ou senha incorretos."). O mesmo princípio aparece na recuperação de senha ("se o e-mail existir...") e nos 404 (em vez de 403) do cidadão.
"O que é o catálogo público de serviços (/servicos/)?"
Bruno: a view catalogo_servicos (views_root.py) é pública e mostra as categorias com seus serviços ativos. Usa a mesma função db.listar_categorias_com_servicos() da home, renderizando public/catalogo_servicos.html.
"Como funciona a paginação das listas de chamados?"
Rafael/Bruno: o SQL já traz só a "fatia" da página (LIMIT %s OFFSET %s) e o total; a função db.paginar(linhas, pagina, por_pagina=15, total_count=...) (db/pagination.py) devolve um objeto de página compatível com o template (15 por página).
"O que faz o helper fetch_all / fetch_one do db/_shared.py?"
Rafael: é um atalho para as listas simples: executa o SELECT e converte cada linha num objeto (SimpleNamespace) a partir de uma lista de nomes de campo, já criando o atributo pk. Evita repetir o with connection.cursor() + laço em cada função de listagem (catálogo, colaboradores, notificações). O SQL continua sendo escrito à mão.
"Qual a diferença entre 'adicionar observação' e 'mudar status'?"
Rafael: os dois são INSERT em historico_chamado. Mudar status insere com um novo id_status (vira o status atual). Observação repete o status atual e preenche o campo observacao — não muda o status, só adiciona uma anotação ao histórico.
"Como garantir que dois chamados abertos ao mesmo tempo não peguem o mesmo protocolo?"
Rafael: o protocolo vem da stored function fn_proximo_protocolo(ano), que faz INSERT INTO protocolo_seq ... ON CONFLICT (ano) DO UPDATE SET ultimo_numero = ... + 1 RETURNING dentro de um LOOP — atômico no banco. Bruno: ainda há um for de retry em criar_novo_chamado que, se um IntegrityError de protocolo acontecer, gera outro número e tenta de novo, tudo em transaction.atomic().
"Onde está a regra de que o colaborador (COL) não mexe em chamado encerrado?"
Bruno: no Python, em views_equipe: no GET, bloqueia_status_col = (perfil == 'COL' and sigla in ('CO','CA')) esconde o form; no POST, equipe_chamado_status reforça com raise Http404() se um COL tentar mexer em CO/CA. Só o GES passa.
"O que acontece se alguém tentar dar UPDATE direto no histórico pelo banco?"
Rafael: o trigger fn_historico_sem_update (04_rules.sql) dá RAISE EXCEPTION — histórico é imutável. DELETE em histórico/chamado também é bloqueado, a não ser dentro do fluxo oficial de exclusão (a aplicação seta set_config('portal.excluindo','true')).
"Por que usar transaction.atomic() ao excluir um chamado?"
Rafael: a exclusão apaga vários "filhos" (fotos, históricos, notificações) e o chamado. Com a transação, ou tudo é apagado, ou nada (se der erro no meio, desfaz) — o banco nunca fica num estado parcial. A mesma flag portal.excluindo destrava os triggers de proteção só durante essa transação.
"Como os formulários são protegidos contra CSRF?"
Yuuki/Bruno: todo <form method="post"> tem {% csrf_token %}, e o CsrfViewMiddleware do Django valida o token em cada POST. Sem o token válido, o Django recusa a requisição — protege contra envio forjado de outro site.
"Onde o número do sininho de notificações é calculado?"
Bruno: num context processor (context_processors.py, registrado no settings) que roda em toda página e injeta notif_count — um SELECT COUNT(*) das notificações não lidas e não arquivadas do usuário. Yuuki: o base.html mostra esse número no sininho do topo.
"Como é feito o controle de acesso por perfil (CID/COL/GES)?"
Bruno: por decorators em cima das views: @perfis("GES"), @perfis("COL","GES"), @perfis("CID"), @autenticado, @anonimo (decorators.py). Quem não tem o perfil é redirecionado com mensagem de "Sem permissão". O middleware carrega o usuário em request.portal_user a cada requisição.
"Quantas tabelas, triggers e views o banco tem?"
Rafael: 14 tabelas no 01_schema.sql (11 entidades do DER + configuracao_semaforo, protocolo_seq e banner_publicacao) — todas managed=False (criadas pelo SQL, espelhadas pelos models). 10 funções PL/pgSQL: 1 stored function (fn_proximo_protocolo) + 9 funções de trigger. 9 triggers: 3 de negócio (Trigger 1/2A/2B em 03_functions_triggers.sql) + 6 de integridade (04_rules.sql: foto/observação em encerrado, avaliação imutável, histórico/chamado sem UPDATE/DELETE). 1 view (vw_estatisticas_chamados).
